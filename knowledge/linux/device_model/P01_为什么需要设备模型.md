# 第1章\_为什么需要设备模型

没有统一模型时，每条总线都要自行维护设备集合、驱动集合、匹配、生命周期、用户空间呈现和电源顺序；设备出现先于驱动、驱动出现先于设备、热拔出和 probe 失败又产生不同分支。问题不是“缺少一个 `struct device`”，而是缺少一套能让对象在任意到达顺序下收敛到正确绑定状态的协议。

> **核心矛盾：** 总线拥有发现与匹配规则，驱动拥有操作能力，设备拥有身份和生命周期；三者必须松耦合，又要在注册、删除和失败时保持一致。

Driver Core 因而统一承担对象注册、集合遍历、匹配调度、绑定状态、引用计数、sysfs/uevent 与 PM 依赖，而具体总线仍提供 `match/probe/remove/uevent` 等策略。
