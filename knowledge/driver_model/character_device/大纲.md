# 第1章\_字符设备专题大纲

## 1.1\_专题定位

字符设备不是“申请设备号再填一张 `file_operations` 表”这么简单。它把用户空间文件描述符、VFS、字符设备映射、驱动对象和真实硬件连接起来。本专题只维护这条通用主线；GPIO、设备树、DMA 和具体 SoC 实验分别链接到其权威位置，不在这里重复展开。

```mermaid
flowchart LR
    U[用户系统调用] --> V[VFS 与 struct file]
    V --> M[dev_t 到 cdev 的映射]
    M --> F[file_operations]
    F --> D[设备对象与驱动状态]
    D --> H[硬件或软件数据源]
```

## 1.2\_阅读阶段

### 1.2.1\_阶段一\_先建立对象模型

1. [字符设备最小模型](P01_字符设备最小模型.md)：分清设备号、设备节点、`cdev`、`inode`、`file` 与 `file_operations`。
2. [设备号、注册与设备节点](P02_设备号_注册与设备节点.md)：理解“内核可分派”和“用户可访问”是两件事。
3. [打开路径与文件操作](P03_打开路径与文件操作.md)：沿 `open()` 建立 `inode->i_cdev`、`file->f_op` 与 `private_data` 的关系。

### 1.2.2\_阶段二\_处理并发和对象生命期

4. [文件操作并发与生命周期](P04_文件操作并发与生命周期.md)：区分设备级、打开实例级和 VMA 级状态，设计阻塞、唤醒、热拔出与释放路径。
5. 通用依赖：[等待队列](../../linux/waiting_notification/P01_等待队列.md)、[生命周期集成专题](../../linux/object_lifetime/integration/大纲.md)。

### 1.2.3\_阶段三\_落地与验证

6. [字符设备驱动模板](P05_字符设备驱动模板.md)：从最小模板进入多实例和阻塞 I/O 模板。
7. [构建运行与注册场景](P06_构建运行与注册场景.md)：验证动态/静态设备号与自动/手工节点的组合。
8. [常见故障与排查](P07_常见故障与排查.md)：根据节点类型、设备号、模块引用和日志定位问题。

## 1.3\_边界与延伸

- i.MX6ULL 的 GPIO、设备树和 LED 组合属于[平台实验](../../../labs/platforms/nxp/imx6ull/drivers/P01_LED点灯/P03_字符设备_LED与设备树完整实验.md)。
- `poll` 的等待条件先学习[等待队列](../../linux/waiting_notification/P01_等待队列.md)，再回到字符设备并发章节理解接口交叉。
- `mmap` 会让内存对象越过 fd 的生命期，应结合对象生命周期专题阅读。
- DMA、IOMMU、IRQ 和零拷贝不是字符设备注册机制的组成部分。历史混合笔记已迁至[调查材料](../../../research/investigations/character_device/字符设备扩展接口历史草稿.md)，只用于追溯，不作为推荐实现。

## 1.4\_学习完成标准

读完后应能独立回答：

1. `register_chrdev_region()`、`cdev_add()` 和 `device_create()` 各自建立了哪一层关系？
2. 第一次打开设备节点时，VFS 怎样找到驱动的 `open()`？
3. `inode->i_cdev` 与 `file->private_data` 分别适合定位什么对象？
4. 多个 fd、`dup()`、`fork()` 和 `mmap()` 会共享哪些状态？
5. 初始化失败、设备移除和模块卸载时，资源应按什么顺序回滚？
