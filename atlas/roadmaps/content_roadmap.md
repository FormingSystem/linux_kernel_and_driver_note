# 第1章\_知识库建设路线图

## 1.1\_路线图用途

本路线图记录知识库覆盖情况和建设优先级，不承诺时间。完成状态以正式文档、有效链接和质量检查共同判断，而不是以目录是否存在判断。

## 1.2\_当前基线

| 领域 | 状态 | 说明 |
| --- | --- | --- |
| Linux 通用机制 | 持续建设 | 已有数据结构、并发、生命周期、设备模型、时间、I/O 和错误处理专题 |
| 驱动模型 | 持续建设 | 已有字符设备、设备树、GPIO、Platform 和 Input 内容 |
| 内核子系统 | 起步 | 中断内容较完整，日志和内核运行基础已有入口，其他子系统待建 |
| 系统软件 | 持续建设 | Buildroot 内容较系统，U-Boot、Kconfig 和链接脚本已有基础材料 |
| 平台与实验 | 起步 | 以 i.MX6ULL 为主，含 RK3566 编译环境记录 |
| 工程方法与项目 | 待建设 | 尚未形成独立的工程方法文档和综合项目入口 |
| 源码研究 | 待整理 | 已有源码阅读材料和调查入口，但索引、版本边界和证据链需补齐 |

## 1.3\_近期优先级\_整理现有资产

1. 为现有专题补齐稳定元数据、入口说明和明确的阅读顺序。
2. 区分稳定知识、平台操作记录、实验和特定版本源码证据，迁移错位内容时同步维护链接。
3. 为实验补齐环境、前置知识、预期结果、实际结果、失败现象和清理方法。
4. 为源码阅读材料记录 Linux 版本、文件路径和提交或标签。
5. 将 atlas 的地图、路线和索引纳入每次结构变更的同步检查。

## 1.4\_中期优先级\_补齐核心领域

- 基础知识：计算机体系结构、C 语言内存模型、操作系统进程与虚拟内存、算法复杂度。
- Linux 通用机制：执行上下文、内存管理、等待与通知、内存序、统一 I/O 模型。
- 内核子系统：调度、内存管理、VFS、网络、块层、电源管理和 tracing。
- 驱动模型：I2C、SPI、UART、PWM、DMA、网络设备、块设备和电源管理。
- 工程方法：内核构建、调试、测试、性能分析、BSP 移植和发布流程。

新主题先确定权威正文归属，再由路线链接；不得为某条路线复制一份知识正文。

## 1.5\_长期优先级\_形成闭环

1. 为关键机制建立“知识正文—实验—源码证据—问题调查”的双向链接。
2. 建设至少一个组合启动链、设备树、驱动、根文件系统和调试方法的完整项目。
3. 扩展 ARM64、RISC-V 和 x86 平台差异，同时保持通用原理与平台实现分离。
4. 使用出版清单编排专题书籍或文章，并由构建流程生成连续章节和稳定锚点。
5. 将孤立文档、失效链接、重复 ID、源码版本信息和实验入口纳入持续质量检查。

## 1.6\_完成定义

一个领域可以标记为“已形成基础闭环”，至少需要：

- 有一篇说明边界、核心对象和关系的权威入口文档。
- 有可执行的学习顺序，并明确前置知识和阶段验收。
- 关键结论至少有实验或特定版本源码证据之一。
- 文档具有合规元数据，内部链接有效，且不与其他领域重复保存正文。
- 已运行元数据、命名、标题、链接和 Git 空白检查。
