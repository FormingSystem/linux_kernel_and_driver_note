# kref 引用计数机制章节大纲

## 第 1 章：kref 要解决什么问题

核心不是“计数器怎么加减”，而是 **内核对象生命周期怎么被多个执行路径安全共享**。

重点讲：

1. 为什么裸指针共享会导致 use-after-free。
2. 为什么引用计数解决的是 **生命周期问题**，不是并发互斥问题。
3. kref 适合“一个对象被多个线程、回调、队列、子系统持有”的场景。
4. kref 不负责对象字段保护；字段一致性仍然靠 mutex/spinlock/RCU 等机制。

官方文档直接说：如果对象会在多个地方被使用和传递，却没有引用计数，代码几乎肯定是坏的；kref 就是给这类对象加引用计数的机制。([内核网][1])

---

## 第 2 章：源码入口与结构定义

源码阅读入口：

```c
include/linux/kref.h
include/linux/refcount.h
```

本章重点：

1. `struct kref` 的本质。
2. `kref` 为什么嵌入到业务对象内部。
3. `kref` 和 `container_of()` 的配合关系。
4. `kref` 不是单独分配的对象，而是业务对象生命周期的一部分。

现代内核中 `struct kref` 内部包的是 `refcount_t`，不是简单裸 `atomic_t`；`kref_init()` 设置初始引用计数为 1，`kref_get()` 通过 `refcount_inc()` 增加引用，`kref_put()` 通过 `refcount_dec_and_test()` 递减并在归零时调用 release 回调。([GitHub][2])

推荐讲解模板：

```c
struct my_obj {
	struct kref ref;
	/* real object fields */
};
```

## 第 3 章：kref 生命周期状态机

这一章建议画成状态机：

```text
allocated
   |
   v
kref_init() -> refcount = 1
   |
   +-- kref_get() -> refcount++
   |
   +-- kref_put() -> refcount--
              |
              +-- refcount != 0 : object still alive
              |
              +-- refcount == 0 : release(kref) -> free object
```

重点不是 API，而是所有权语义：

1. `kref_init()` 创建初始引用。
2. 每个长期保存指针的一方必须拥有引用。
3. 每个拥有引用的一方退出时必须 `kref_put()`。
4. 最后一个 `kref_put()` 触发对象释放。
5. release 回调是对象销毁点。

官方文档明确：初始化后引用计数为 1；使用完成后调用 `kref_put()`；如果这是最后一个引用，release 例程会被调用。([内核网][1])

## 第 4 章：kref 三条核心规则

这是本章最重要的内容，应作为 P0 重点。

### 规则 1：非临时拷贝指针之前，必须先 get

错误模型：

```c
pass_to_thread(obj);
kref_get(&obj->ref);   /* 错：交出去之后再 get */
```

正确模型：

```c
kref_get(&obj->ref);
pass_to_thread(obj);
```

官方文档特别强调：`before` 很重要，不要在 handoff 之后再 `kref_get()`。([内核网][1])

### 规则 2：使用完指针必须 put

```c
kref_put(&obj->ref, obj_release);
```

如果最后一个引用释放，release 被调用。([内核网][1])

### 规则 3：没有现成有效引用时，lookup + get 必须被保护

也就是：
你不能从链表、哈希表、idr/xarray 等结构里拿到裸指针后，直接 `kref_get()`，除非你能保证对象在 get 过程中不会被释放。

官方文档说，如果代码试图在没有已持有有效指针的情况下获取引用，必须序列化访问，确保 `kref_put()` 不会在 `kref_get()` 期间发生，并且结构体在 `kref_get()` 期间保持有效。([内核网][1])

## 第 5 章：基础 API 源码逐行讲解

本章按源码顺序讲：

```c
KREF_INIT(n)
kref_init()
kref_read()
kref_get()
kref_put()
```

重点解释：

1. `KREF_INIT(n)` 用于静态初始化。
2. `kref_init()` 用于动态对象初始化，初始值为 1。
3. `kref_read()` 只能读计数，不能作为生命周期判断依据。
4. `kref_get()` 只能在已经有有效引用的前提下使用。
5. `kref_put()` 归零时调用 release，并返回 1；否则返回 0。

`kref_put()` 的返回值只能可靠说明“本次调用确实释放了对象”；如果返回 0，不能据此推断对象之后仍然存在，因为别的线程可能马上释放它。源码注释也强调了这个返回值语义。([GitHub][2])

## 第 6 章：release 回调与 container_of 模式

本章讲标准析构写法：

```c
static void my_obj_release(struct kref *ref)
{
	struct my_obj *obj = container_of(ref, struct my_obj, ref);

	kfree(obj);
}
```

重点：

1. release 参数是 `struct kref *`，不是业务对象指针。
2. 必须用 `container_of()` 找回外层对象。
3. release 是最后引用释放点，不能再把对象交出去。
4. release 可以做复合清理：从链表删除、释放内存、释放子资源。
5. 不能把 `kfree` 直接当 release 传入；release 需要知道外层对象类型。

源码注释明确要求 release 是清理对象的函数指针，`kref_put()` 调用者不能传 NULL 或直接传 `kfree()`。([GitHub][2])

---

## 第 7 章：handoff 所有权转移模型

这一章专门讲“引用转移”，否则很容易写出多余 get/put 或漏 put。

典型场景：

```c
/* 当前线程持有一个引用 */
enqueue(obj);
/* 从这里开始不再访问 obj，引用所有权转移给队列 */
```

重点：

1. 有时不需要 `kref_get()` 再 `kref_put()`。
2. 如果当前引用直接转移给队列、workqueue、线程，则不额外加减。
3. 但转移后当前路径不能再访问对象。
4. 代码中必须注释清楚所有权转移边界。

官方文档也给出了这个优化点：如果只是把自己当前持有的引用交给队列，就不要做“额外 get 再 put”的傻操作，并建议注释“之后不要再碰这个对象”。([内核网][1])

---

## 第 8 章：lookup 场景与 kref_get_unless_zero()

本章是 kref 的难点。

要讲清楚两个问题：

1. 从容器里查对象时，查到的是裸指针。
2. 裸指针不代表你已经拥有引用。

`kref_get_unless_zero()` 的作用是：

```c
if (!kref_get_unless_zero(&obj->ref))
	obj = NULL;
```

它只在引用计数非 0 时增加引用；如果已经是 0，说明对象正在释放或已经不可获得，返回失败。

源码中 `kref_get_unless_zero()` 使用 `refcount_inc_not_zero()`，并且带 `__must_check`，也就是返回值必须检查。([GitHub][2])

本章必须强调：

```c
obj = lookup();
kref_get_unless_zero(&obj->ref);  /* 仍然可能错 */
```

如果 lookup 自身没有锁/RCU保护，`obj` 可能已经被释放。官方文档明确说，`kref_get_unless_zero()` 必须包在同一个查找临界区里，否则可能引用已经释放的内存；并且不能忽略返回值。([内核网][1])

## 第 9 章：kref 与锁的组合

本章讲两个问题：

1. 对象生命周期用 kref。
2. 对象集合关系用锁保护。

重点 API：

```c
kref_put_mutex()
kref_put_lock()
```

源码里 `kref_put_mutex()` 使用 `refcount_dec_and_mutex_lock()`，归零时在持有 mutex 的情况下调用 release；`kref_put_lock()` 类似，只是使用 spinlock。([GitHub][2])

要讲清楚：

1. 这些 API 不是为了保护引用计数本身。
2. 它们是为了保护“最后一个 put + 从全局结构删除 + release”这个复合过程。
3. release 回调会在锁持有状态下被调用。
4. release 必须知道自己要释放锁，否则会死锁或锁不平衡。

---

## 第 10 章：kref 与 RCU

本章讲高级组合：

```text
RCU 保护查找路径
kref 保护对象生命周期
kfree_rcu() 延迟释放内存
```

重点：

1. RCU 读侧可以找到对象，但不自动拥有引用。
2. 读侧查到对象后，需要 `kref_get_unless_zero()` 尝试取得引用。
3. release 路径从 RCU 链表删除对象。
4. 最终释放要用 `kfree_rcu()` 或在 `synchronize_rcu()` 后释放。
5. `struct kref` 所在内存必须在 RCU grace period 结束前保持有效。

官方文档的 RCU 示例明确指出：release 后，包含 `struct kref` 的内存还需要在一个 RCU grace period 内保持有效，可以通过 `kfree_rcu()` 或 `synchronize_rcu()` 实现。([内核网][1])

---

## 第 11 章：kref 不是 kobject，但 kobject 依赖引用计数思想

本章用于边界区分。

重点：

1. `kref` 是通用引用计数小工具。
2. `kobject` 是设备模型对象，有 sysfs、层次结构、uevent 等语义。
3. kref 可以独立用于任意内核对象。
4. 不要为了引用计数强行引入 kobject。
5. kobject 的生命周期设计可以作为复杂对象参考，但 kref 本身更轻量。

这一章建议只做对比，不展开 kobject 全体系，否则章节会跑偏。

## 第 12 章：典型错误模式

建议列成检查清单。

### 错误 1：handoff 后再 get

```c
queue_work(obj);
kref_get(&obj->ref);
```

### 错误 2：lookup 后无保护 get

```c
obj = list_first_entry(...);
kref_get(&obj->ref);
```

### 错误 3：put 后继续访问对象

```c
kref_put(&obj->ref, obj_release);
obj->state = DEAD;  /* 错：obj 可能已经释放 */
```

### 错误 4：release 里只 kfree，不做脱链

如果对象还挂在全局 list/hash/tree 里，只 `kfree()` 会留下悬挂指针。

### 错误 5：把 kref 当锁

```c
kref_get(&obj->ref);
obj->field++;  /* 并发安全？不一定 */
```

kref 只保证对象没被释放，不保证字段修改互斥。

## 第 13 章：工程模板

本章给一个标准模板：

```c
struct my_obj {
	struct kref ref;
	struct mutex lock;
	struct list_head node;
	int state;
};

static void my_obj_release(struct kref *ref)
{
	struct my_obj *obj = container_of(ref, struct my_obj, ref);

	kfree(obj);
}

static struct my_obj *my_obj_get(struct my_obj *obj)
{
	kref_get(&obj->ref);
	return obj;
}

static void my_obj_put(struct my_obj *obj)
{
	kref_put(&obj->ref, my_obj_release);
}
```

然后扩展：

1. 单对象引用模板。
2. list/hash 查找模板。
3. RCU 查找模板。
4. workqueue handoff 模板。
5. error path 清理模板。

## 第 14 章：源码阅读实验

建议实验按这个顺序：

1. 写一个最小 kref 对象：alloc/init/get/put/release。
2. 模拟两个线程持有同一对象。
3. 故意 handoff 后再 get，分析 use-after-free 窗口。
4. 写 list + mutex + kref_get 的查找模型。
5. 改成 `kref_get_unless_zero()`。
6. 改成 RCU + `kfree_rcu()`。
7. 用 KASAN/KCSAN/lockdep 辅助观察错误。

## 第 15 章：最终验收标准

学完这一章，至少要能做到：

1. 能解释 kref 保护的是生命周期，不是数据一致性。
2. 能手写 `struct kref` 嵌入对象模板。
3. 能解释 `kref_init()` 为什么初始值是 1。
4. 能解释为什么 handoff 前必须先 get。
5. 能解释为什么 `kref_get_unless_zero()` 必须检查返回值。
6. 能写出 list/hash lookup + kref 的正确锁模型。
7. 能写出 RCU lookup + kref 的正确释放模型。
8. 能判断 `kref_put()` 返回 0 时不能继续假设对象仍然存在。
9. 能识别 put 后继续访问、无保护 lookup、release 里不脱链这几类典型 bug。

# 本章 P0 主线

一句话：**kref 不是“一个计数器 API”，而是 Linux 内核对象生命周期协议。**

讲的时候不要围绕 `++/--` 展开，而要围绕这条主线：

```text
谁持有对象？
谁负责 get？
谁负责 put？
对象从哪里可被查到？
最后一个 put 时，如何从全局结构移除？
内存什么时候真正释放？
```

这个大纲才符合 Linux kernel 6.12 源码视角。
